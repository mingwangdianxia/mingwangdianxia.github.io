系统架构的演化过程

几乎所有的大型互联网网站都是从小网站开始，逐渐发展起来的，这也正好与系统架构的发展演化过程对应。

发展的终点：

  • 三高（高并发、高可用、高性能）
  • 海量数据
  • 用户分布广泛
  • 需求更新频繁，产品快速迭代
  • …

在这篇笔记中《系统架构演化》详细说明了系统演化的历程，在系统的更新迭代过程中，突出设计架构的目的：解决以往业务中的痛点

单应用系统组合问题

  • 系统间通信困难
  • 系统的内部复杂度对外暴露
  • 系统间的调用关系复杂
  • 过度的重复建设
  • 大一统的架构设计

分布式理论基础

ACID

传统的事务遵循定理

  • A：原子性：所有操作要么全部做完，要么都不做
  • C：一致性：数据库要一直处于一致的状态，事务的运行不会改变数据库原本的一致性约束
  • I：独立性：并发的事务之间不会互相影响
  • D：持久性：指一旦事务提交后，数据会永久保存

CAP

分布式条件下是基于异步网络，传统的事务定理不再适合，因此出现CAP，三选二，因为三个条件不能同时满足（AP或CP）分布式架构的时候必须做出取舍。 一致性和可用性之间取一个平衡。多余大多数web应用，其实并不需要强一致性

  • C：一致性：各个节点的数据要一致
  • A：可用性：单一节点出现故障，整个系统要能正常提供服务
  • P：分区容错性：能够容忍节点之间的通讯故障

CAP理论的核心是：一个分布式系统不可能同时很好的满足一致性，可用性和分区容错性这三个需求， 最多只能同时较好的满足两个。 因此，根据 CAP原理将NoSQL数据库分成了满足CA原则、满足CP原则和满足AP原则三大类：

  • CA - 单点集群，满足一致性，可用性的系统，通常在可扩展性上不太强大。

  • CP - 满足一致性，分区容忍必的系统，通常性能不是特别高。

  • AP - 满足可用性，分区容忍性的系统，通常可能对一致性要求低一些。

BASE

BASE是为了解决关系数据库强一致性引起的可用性降低提出的解决方案

  • BA：基本可用
  • S：软状态
  • W：最终一致性

它的思想是通过让系统放松对某一时刻数据一致性的要求来换取系统整体伸缩性和性能上改观

微服务

微服务是一种架构风格，一个大型复杂软件应用由一个或多个微服务组成。系统中的各个微服务可被独立部署，各个微服务之间是松耦合的。每个微服务仅关注于完成一件任务并很好地完成该任务。在所有情况下，每个任务代表着一个小的业务能力

微服务，从本质意义上看，还是 SOA 架构。但内涵有所不同，微服务并不绑定某种特殊的技术，在一个微服务的系统中，可以有 Java 编写的服务，也可以有 Python编写的服务，他们是靠Restful架构风格统一成一个系统的。所以微服务本身与具体技术实现无关，扩展性强。

微服务特点

  • 构建基于业务、可重用、职责明确
  • 统一的通信标准，轻量级的通信协议
  • 可独立运行，独立存储由独立团队进行维护

微服务架构问题

  • 跨进程通信
  • 较高的响应延迟
  • 运维难度和成本上升
  • 组织架构层面调整
  • 服务间的集成测试困难

微服务架构设计原则

  • 确定微服务的划分原则，没有统一的标准，适合就是最好的，不过划分还是有几项准则：
    • 单一职责原则：一个微服务制作好一件事，体现高内聚，低耦合
    • 服务依赖原则：避免服务间的循环引用
  • 确定微服务模块的使命
  • 微服务独立的数据存储
  • 服务间的通信采用聚合器模式

微服务使用场景

  • 新规划的大型业务系统
  • 敏捷的小团队系统
  • 历史的大型留存业务系统
