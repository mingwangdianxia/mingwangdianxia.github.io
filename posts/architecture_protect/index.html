微服务的系统保护

雪崩

某一节点长时间处理导致应用请求积压崩溃的现象，出现原因：瞬间大流量和微服务响应慢

解决方案：

  • 限流：控制请求的流入，让流量有序的进入应用，保证在一个可控的范围
  • 服务降级与熔断：当应用处理慢超过预定值时候，返回请求预定的异常

Sentinel

从流量控制、熔断降级、系统负载保护等多个维度保护服务的稳定性

注意
sentinel-dashboard基于懒加载模式，如果找不到对应的uri，需要先访问下这个功能后才会出现

sentinel通信与降级原理

Sentine Core 在初始化的时候，通过 application.yml 参数中指定的 Dashboard 的 IP地址，会主动向 dashboard 发起连接的请求。

限流降级的规则配置

实现限流的核心就是统计单位时间某个接口的访问量，常见的算法有：计数器算法，令牌桶算法，漏桶算法，滑动窗口算法

滑动窗口算法：将整段时间均分后独立计数再汇总统计

流控名词：

  • 资源名：需要流控的URI
  • 针对来源：默认default代表所有来源，可以针对某个微服务或者调用者来单独设置
  • 阈值类型：是QPS（每秒访问数量）还是并发数（线程数）进行流控
  • 单机阈值：具体限流的数值

流控效果：

  • 快速失败：当流量达到阈值之后，直接返回响应并抛出异常
  • 预热（Warm Up）：用于应对瞬时大并发流量冲击。当遇到突发大流量 Warm Up 会缓慢拉升阈值限制，预防系统瞬时崩溃，这期间超出阈值的访问处于队列等待状态，并不会立即抛出异常。
  • 排队等待：采用匀速放行的方式对请求进行处理

流控模式：

  • 直接模式：
  • 关联模式
  • 链路模式

熔断的规则配置

熔断是一种保护机制，当事物的状态达到某种“不可接受”的情况时，便会触发“熔断”

Sentinel Dashboard可以设置三种不同的熔断模式：慢调用比例、异常比例、异常数
